Reading Responses:

1st Article: Why Software Development is a Craft by Doug Bradbury

The author's evidence for Software Engineering to be considered a craft is because of similar generic practices in craftsmanship and Software Engineering:

  • Making: Crafter does most of the work; programmer does most of the work.
  • Material: Better material allows the crafter to make more intricate art; Access to high level programming language allows programmers to create more powerful programs.
  • Tools: Crafting tools; IDEs, compilers, etc
  • Learning: Next project is always better than the last, but worse than the future project. Constantly learning new things about your trade.
    • A new thing I learned from this article was that the Software Craftsmanship crowd is gaining traction.

    My thoughts:

  • Everything is a craft when looked at in the particular angle the author presented, even walking down the street.
  • You can't teach someone to paint like DaVinci, but you can teach someone to code like John Sterling. Craftsmanship shouldn't be able to be learned.

    3rd Article: Programming is Not a Craft by Dan North

    This is the last article I read among the 3 that were assigned, and this article basically stated what I was thinking while reading the first two articles.

    Programmers are: Romantics with big egos. Programmers are: Tasked with performing a task for an end user. End users do not care whether the code is "a work of art"... All the end user wants from the program is that it does what it is supposed to do.

    However, when a true artist or craftsman creates something, the patron is almost always visibly thrilled with the outcome. The patron is interested in the intricate stylistic choices the artist or craftsman decided to implement.

    Standard End Users Are Not Interested.

    This line from the author confused me a little:

    "What makes it a craft is the work above and beyond its basic utility to give it intrinsic, aesthetic beauty."
    It confused me because so far, the author was going at a steady pace and covering all the logical points, but with that quote, he presents his opinion as if it's a dictionary definition.

    My experience working as an IT System Admin has taught be that whatever the CFO wants, goes. My CFO does not care if the SQL code running a office database looks good, all he cares about is if the database is up and running for the appropriate users. As a programmer, if I were to miss the database deadline because I was busy making my code stylistic and beautiful instead of functional, I wouldn't expect to keep my job.

    Sure, I'd be a software craftsman, but I'd also be unemployed.
